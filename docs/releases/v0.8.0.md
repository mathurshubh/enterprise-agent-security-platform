## Overview

Version **0.8.0** introduces a provider-agnostic architecture for the Enterprise Agent Security Platform.

Previous releases tightly coupled the platform to a single Large Language Model (LLM) provider. While functional, this design limited extensibility and required application changes whenever a new model provider was introduced.

The primary objective of this release is to decouple provider selection from the runtime while preserving the platform's Zero Trust architecture. The runtime, authorization pipeline, policy engine, risk engine, detection engine, and response engine remain deterministic and independent of any specific LLM.

With this release, the platform now supports multiple LLM providers through a common abstraction without modifying the core security pipeline.

---

## Design Motivation

Previous releases tightly coupled the platform to a single LLM implementation through the runtime. While sufficient for early development, this approach violated the long-term architectural objective of provider independence.

Enterprise customers increasingly require flexibility in model selection due to regulatory requirements, deployment constraints, cost optimization, and rapid changes in the LLM ecosystem.

Version 0.8 addresses this limitation by introducing a provider abstraction layer that isolates the runtime and deterministic security pipeline from provider-specific implementations.

This architectural change enables new providers to be added without modifying the authorization, policy, detection, risk, response, or runtime components.

# Major Architectural Changes

## EnterpriseAgent Abstraction

A new `EnterpriseAgent` abstraction has been introduced to define the contract for enterprise AI agents.

Responsibilities:

- Accept natural language requests.
- Convert user intent into a validated `ToolInvocation`.
- Delegate provider communication.
- Perform no authorization or security decisions.

The EnterpriseAgent remains an intent parser only.

---

## ProviderAdapter

A new `ProviderAdapter` abstraction defines a common interface for all LLM providers.

Responsibilities:

- Send prompts to a provider.
- Return structured responses.
- Hide provider-specific SDKs and APIs from the rest of the platform.

The runtime is now completely independent of any individual LLM provider.

---

## Ollama Provider

The existing Ollama integration has been refactored behind the new ProviderAdapter interface.

This change preserves existing functionality while aligning the implementation with the new provider architecture.

---

## Gemini Provider

Support has been added for Google's Gemini models through a dedicated provider implementation.

The Gemini provider includes:

- Dedicated service layer
- Provider implementation
- Automated unit tests

Provider integration was validated independently without impacting the deterministic security pipeline.

---

## Provider Factory

A ProviderFactory has been introduced to centralize provider selection.

Provider selection is now configuration-driven rather than hardcoded.

Current supported providers:

- Ollama
- Gemini

Future providers can be added without modifying runtime logic.

---

## Configuration

Provider selection is now externalized through application configuration.

Example:

```text
DEFAULT_PROVIDER=ollama
```

or

```text
DEFAULT_PROVIDER=gemini
```

The runtime no longer contains provider-specific construction logic.

# Security Architecture

A core design principle of the platform remains unchanged:

> The LLM is treated as an untrusted intent parser.

Regardless of the selected provider, the LLM is responsible only for converting natural language into a `ToolInvocation`.

The following components remain deterministic:

- Authentication
- Authorization
- Policy Evaluation
- Risk Assessment
- Detection
- Response
- Tool Execution
- Audit Logging

No provider is capable of influencing security decisions.
