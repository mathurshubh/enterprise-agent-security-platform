# ADR-005: Adopt a Tool Registry for Controlled Tool Execution

**Status:** Accepted

**Date:** 2026-06-20

**Authors:**
- Shubhankar Mathur

---

# Context

Enterprise AI agents interact with external systems by invoking tools.

Without a centralized mechanism for defining and managing available tools, AI-generated requests could reference arbitrary or unknown functionality.

Allowing unrestricted tool execution introduces several risks:

- execution of unauthorized code
- inconsistent security enforcement
- privilege escalation
- lack of auditability
- inconsistent tool metadata
- provider-specific behavior
- uncontrolled platform growth

The platform requires a deterministic mechanism for defining which tools exist and which tools may be executed.

---

# Decision

The Enterprise Agent Security Platform adopts a centralized Tool Registry.

Every executable tool must be registered before it can be invoked.

The Tool Registry acts as the authoritative inventory of executable platform capabilities.

Each registered tool exposes metadata describing its:

- identity
- description
- supported parameters
- resource requirements
- execution characteristics

The Runtime Service resolves ToolInvocations through the Tool Registry before execution.

Unregistered tools are rejected before authorization or execution.

---

# Rationale

A centralized Tool Registry separates tool discovery from tool execution.

This provides several architectural benefits.

## Controlled Execution

Only registered tools may participate in the platform.

This prevents arbitrary execution requests generated by LLMs.

---

## Consistent Metadata

All tools expose a common interface and metadata model.

Security services can evaluate requests without provider-specific knowledge.

---

## Extensibility

New tools can be added without modifying the Runtime Service or Enterprise Agents.

Registration becomes the integration point.

---

## Governance

The Tool Registry becomes the authoritative inventory of platform capabilities.

Future releases can associate:

- ownership
- versioning
- capabilities
- risk classifications
- approval requirements

with registered tools.

---

## Auditability

Every executed tool has a stable identity that can be referenced by:

- audit events
- authorization policies
- detections
- metrics
- operational dashboards

---

# Alternatives Considered

## Direct Tool Invocation

Allow Enterprise Agents to execute tools directly by name.

### Rejected

This permits arbitrary execution requests and bypasses centralized governance.

---

## Dynamic Tool Discovery

Allow tools to self-register during runtime without governance.

### Rejected

Dynamic discovery complicates auditing and weakens deterministic control over executable capabilities.

---

## Provider-Specific Tool Definitions

Allow each LLM provider to define available tools independently.

### Rejected

Tool definitions should remain platform-owned rather than provider-owned.

This preserves provider independence.

---

# Consequences

## Positive

- Centralized governance of executable capabilities.
- Consistent tool metadata.
- Simplified authorization.
- Improved auditability.
- Supports future tool ecosystem growth.

## Negative

- Every new tool requires explicit registration.
- Additional operational management of the registry.
- Registry availability becomes important for tool execution.

## Risks

- Poor registry governance could lead to stale metadata.
- Large tool ecosystems may require categorization and lifecycle management.

---

# Security Considerations

The Tool Registry defines the boundary between AI-generated intent and executable platform capabilities.

Enterprise Agents never execute tools directly.

Every ToolInvocation is first resolved against the Tool Registry.

Only registered tools continue through the deterministic security pipeline.

The Tool Registry itself does not authorize execution.

Authorization remains the responsibility of dedicated security services.

# Notes

The Tool Registry governs executable capabilities rather than authorization.

Registration determines whether a tool exists within the platform.

Authorization determines whether a specific ToolInvocation may execute.
