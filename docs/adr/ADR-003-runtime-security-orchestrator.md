# ADR-003: Establish the Runtime Service as the Security Orchestrator

**Status:** Accepted

**Date:** 2026-06-20

**Authors:**
- Shubhankar Mathur

---

# Context

Enterprise AI agents generate requests that may result in the execution of privileged tools and access to sensitive enterprise resources.

Without a centralized orchestration layer, individual agents or tool implementations would become responsible for security enforcement.

This would lead to:

- inconsistent authorization
- duplicated security logic
- fragmented auditing
- inconsistent policy enforcement
- increased attack surface

The platform requires a single trusted component responsible for coordinating deterministic security decisions before tool execution.

---

# Decision

The Runtime Service is designated as the central security orchestrator for all tool invocations.

The Runtime Service is responsible for coordinating the platform's deterministic security pipeline.

Every `ToolInvocation` generated by an Enterprise Agent must pass through the Runtime Service before any tool execution occurs.

The Runtime Service orchestrates interactions with:

- Authorization Service
- Policy Engine
- Session Service
- Detection Service
- Risk Service
- Response Service
- Tool Registry

The Runtime Service coordinates these components but does not make independent security decisions.

Each security service remains responsible for its own deterministic function.

---

# Rationale

Centralizing orchestration within the Runtime Service provides several architectural advantages.

## Separation of Responsibilities

Each security component remains focused on a single responsibility while the Runtime Service coordinates their execution.

## Consistent Security Enforcement

Every request follows the same deterministic processing pipeline regardless of:

- LLM provider
- Enterprise Agent
- Tool
- User

## Reduced Attack Surface

Tool execution cannot bypass authorization or policy evaluation because every request follows the same orchestration path.

## Extensibility

New security services can be introduced into the pipeline without modifying Enterprise Agents or individual tools.

---

# Alternatives Considered

## Agent-Managed Tool Execution

Allow Enterprise Agents to invoke tools directly.

### Rejected

This couples AI inference with security enforcement and allows different agents to implement security inconsistently.

It also increases the likelihood of authorization bypasses.

---

## Tool-Level Authorization

Allow each tool to implement its own authorization logic.

### Rejected

This duplicates security logic across the codebase and makes consistent policy enforcement difficult.

Security controls should remain centralized.

---

## Distributed Security Decisions

Allow multiple components to independently determine whether execution should continue.

### Rejected

This complicates reasoning about the security model and makes auditing significantly more difficult.

A centralized orchestration layer provides a clearer execution model.

---

# Consequences

## Positive

- Single orchestration point for security.
- Consistent request processing.
- Simplified auditing.
- Easier testing.
- Improved maintainability.
- Supports future security services without changing agents.

## Negative

- Runtime Service becomes a critical platform component.
- Additional orchestration introduces some processing overhead.
- Service availability becomes important for the overall platform.

## Risks

- Runtime Service may become overly complex if business logic is added.
- Excessive coupling may occur if component responsibilities are not carefully maintained.

---

# Security Considerations

The Runtime Service is trusted infrastructure.

It never delegates security decisions to Enterprise Agents or LLM providers.

Instead, it coordinates deterministic security services while preserving explicit trust boundaries.

The Runtime Service also prevents direct communication between Enterprise Agents and enterprise tools, ensuring that every request is subject to the complete security pipeline.

# Notes

The Runtime Service is intentionally designed as an orchestration layer rather than a decision-making component.

Authorization, policy evaluation, detection, risk assessment, and response remain the responsibility of dedicated deterministic services.

This separation preserves the platform's Zero Trust architecture while enabling future expansion of the security pipeline without affecting Enterprise Agents or tool implementations.
