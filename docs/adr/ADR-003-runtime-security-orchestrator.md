# ADR-003: Establish the Runtime Layer as the Security Orchestrator

**Status:** Accepted

**Date:** 2026-06-20

**Authors:**
- Shubhankar Mathur

---

# Context

Enterprise Agents generate requests that may result in the execution of privileged tools and access to sensitive enterprise resources.

Without a centralized orchestration layer, individual agents or tool implementations would become responsible for security enforcement.

This would lead to:

- inconsistent authorization
- duplicated security logic
- fragmented auditing
- inconsistent policy enforcement
- increased attack surface

The platform requires a trusted runtime layer responsible for coordinating deterministic security decisions before tool execution.

---

# Decision

The runtime layer is designated as the central security orchestrator for all `ToolInvocation` processing.

The `RuntimeService` is responsible for coordinating the platform's deterministic security pipeline and producing a `RuntimeResult`.

Every `ToolInvocation` generated by an Enterprise Agent must pass through the runtime layer before any tool execution occurs.

The current deterministic runtime evaluation coordinates:

- Authorization Service
- Policy Engine
- Session Service
- Detection Service
- Risk Service
- Response Service

The surrounding runtime composition layer coordinates provider invocation, `ToolInvocation` handling, `RuntimeResult` evaluation, and final `BaseTool` resolution through the Tool Registry.

The Runtime Service coordinates security components but does not make independent security decisions.

Each security service remains responsible for its own deterministic function.

---

# Rationale

Centralizing orchestration within the runtime layer provides several architectural advantages.

## Separation of Responsibilities

Each security component remains focused on a single responsibility while the runtime layer coordinates their execution.

## Consistent Security Enforcement

Every request follows the same deterministic processing pipeline regardless of:

- LLM provider
- Enterprise Agent
- Tool
- User

## Reduced Attack Surface

Tool execution cannot bypass authorization, policy evaluation, detection, risk assessment, or response selection because every request follows the same orchestration path before Tool Registry resolution.

## Extensibility

New security services can be introduced into the pipeline without modifying Enterprise Agents, providers, or individual tools.

---

# Alternatives Considered

## Agent-Managed Tool Execution

Allow Enterprise Agents to invoke tools directly.

### Rejected

This couples AI inference with security enforcement and allows different agents to implement security inconsistently.

It also increases the likelihood of authorization bypasses.

---

## Tool-Level Authorization

Allow each tool to implement its own authorization logic.

### Rejected

This duplicates security logic across the codebase and makes consistent policy enforcement difficult.

Security controls should remain centralized.

---

## Distributed Security Decisions

Allow multiple components to independently determine whether execution should continue.

### Rejected

This complicates reasoning about the security model and makes auditing significantly more difficult.

A centralized runtime orchestration layer provides a clearer execution model.

---

# Consequences

## Positive

- Single orchestration point for security.
- Consistent request processing.
- Simplified auditing.
- Easier testing.
- Improved maintainability.
- Supports future security services without changing agents.

## Negative

- Runtime layer becomes a critical platform component.
- Additional orchestration introduces some processing overhead.
- Service availability becomes important for the overall platform.

## Risks

- Runtime layer may become overly complex if business logic is added.
- Excessive coupling may occur if component responsibilities are not carefully maintained.

---

# Security Considerations

The runtime layer is trusted infrastructure.

It never delegates security decisions to Enterprise Agents or LLM providers.

Instead, it coordinates deterministic security services while preserving explicit trust boundaries.

The runtime layer also prevents direct communication between Enterprise Agents and enterprise tools. Executable tools are resolved only through the Tool Registry after the deterministic pipeline has produced an allowed `RuntimeResult`.

# Notes

The Runtime Service is intentionally designed as an orchestration layer rather than a policy decision-making component.

Authorization, policy evaluation, detection, risk assessment, and response remain the responsibility of dedicated deterministic services.

This separation preserves the platform's Zero Trust architecture while enabling future expansion of the security pipeline without affecting Enterprise Agents, provider adapters, or tool implementations.
