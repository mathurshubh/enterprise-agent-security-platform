# ADR-002: Treat the LLM as an Untrusted Intent Parser

**Status:** Accepted

**Date:** 2026-06-20

**Authors:**
- Shubhankar Mathur

---

# Context

Large Language Models provide powerful natural language understanding capabilities but are fundamentally probabilistic systems.

They may:

- hallucinate
- misunderstand user intent
- generate invalid outputs
- follow malicious instructions
- be influenced by prompt injection
- be affected by indirect prompt injection
- produce inconsistent responses
- return malformed structured data

The Enterprise Agent Security Platform governs enterprise AI agents that may interact with sensitive systems and enterprise resources.

Relying on LLMs to make security decisions would introduce non-deterministic behavior into the platform and significantly increase the attack surface.

The platform therefore requires a clear separation between AI-driven intent interpretation and deterministic security enforcement.

---

# Decision

The platform treats every LLM as an untrusted intent parser.

The LLM is responsible only for converting natural language into structured intent represented by a `ToolInvocation`.

The LLM is **not** responsible for:

- authentication
- authorization
- policy evaluation
- risk assessment
- security decisions
- tool execution
- access control
- trust evaluation

Every `ToolInvocation` produced by the LLM is considered untrusted input.

Before any requested action can execute, it must pass through the deterministic security pipeline.

---

# Rationale

Large Language Models excel at interpreting natural language but cannot provide deterministic security guarantees.

Separating intent interpretation from security enforcement provides several architectural benefits:

- deterministic authorization decisions
- provider-independent security behavior
- improved auditability
- reduced impact of prompt injection attacks
- easier validation and testing
- simpler reasoning about trust boundaries

This separation also allows the platform to support multiple LLM providers without changing its security model.

---

# Alternatives Considered

## Trust the LLM to Make Security Decisions

Allow the LLM to decide whether a requested action should be executed.

### Rejected

LLMs are probabilistic systems and cannot provide deterministic security guarantees.

Prompt injection, hallucinations, and inconsistent reasoning make this approach unsuitable for enterprise security.

---

## Embed Authorization Logic in Prompts

Describe authorization rules within the system prompt and rely on the LLM to enforce them.

### Rejected

Prompt-based authorization is not enforceable.

Security policies must remain deterministic, independently testable, and auditable.

---

## Hybrid Security Model

Allow the LLM to perform preliminary security checks before deterministic validation.

### Rejected

This introduces unnecessary complexity while providing limited security value.

The deterministic security pipeline already performs authoritative validation.

---

# Consequences

## Positive

- Clear separation of responsibilities.
- Consistent security behavior across providers.
- Reduced susceptibility to prompt injection.
- Simplified testing of security controls.
- Supports future provider abstraction.

## Negative

- Every request requires additional validation.
- More platform services participate in request processing.
- Structured output validation becomes mandatory.

## Risks

- Invalid or malformed `ToolInvocation` objects generated by the LLM.
- Increased operational complexity due to multiple validation layers.

---

# Security Considerations

Every LLM output is treated as untrusted until validated.

Security enforcement occurs only after:

1. ToolInvocation validation
2. Authentication
3. Authorization
4. Policy evaluation
5. Detection
6. Risk assessment
7. Response evaluation

The LLM never directly interacts with enterprise tools or protected resources.

Compromise of an LLM provider must not bypass deterministic platform security controls.

# Notes

This decision establishes the fundamental trust boundary between AI inference and enterprise security enforcement.

Future architectural changes should preserve this separation and must not delegate deterministic security decisions to LLMs.
